Un método, múltiples implementaciones

En C++: Un puntero de tipo base (Base*) puede apuntar a cualquier objeto derivado, y al llamar un método virtual, se ejecuta el método del objeto real, no el del puntero.

Tipos de Polimorfismo en C++

⚡

Polimorfismo Estático

Compile-Time
  • El compilador resuelve qué función llamar
  • Más rápido — sin overhead en ejecución
  • Se logra mediante sobrecarga de funciones
  • También con sobrecarga de operadores
void area(int r);         // círculo
void area(int b, int h); // rectángulo
// El compilador elige cuál usar
🔀

Polimorfismo Dinámico

Run-Time
  • Se resuelve en tiempo de ejecución
  • Requiere funciones virtual + herencia
  • Usa punteros o referencias a clase base
  • Mecanismo: vtable (tabla de funciones virtuales)
Animal* a = new Perro();
a->hablar(); // → "Guau" ✓
// No "Animal::hablar()"
Para esta tarea, el polimorfismo que importa es el dinámico (runtime): funciones virtual con herencia de Casilla y GameMode.
TC1030 · Funciones Virtuales
5 / 20

Funciones Virtuales

1
Base declara virtual

Avisa al compilador que esta fn. puede ser sobrescrita

→
2
Hija re-implementa

Misma firma, usa override (recomendado)

→
3
Base* apunta a Hija

Se llama la versión de la hija en runtime

// ── Clase Base ──
class Animal {
public:
  virtual void hablar() {
    cout << "...";
  }
};

// ── Clase Derivada ──
class Perro : public Animal {
public:
  void hablar() override {
    cout << "¡Guau!";
  }
};

// ── main ──
Animal* a = new Perro();
a->hablar(); // → "¡Guau!" ✓
La palabra virtual:
• Solo se escribe en la declaración (.h), no en la implementación (.cpp)
• Una vez virtual en la base, es virtual en todas las hijas
• override es opcional pero detecta errores de firma
Sin virtual: aunque el puntero sea Animal*, si el método NO es virtual, siempre llamará Animal::hablar(), no la versión del hijo. Esto se llama binding estático.
Con virtual: el compilador consulta la vtable del objeto real en tiempo de ejecución. Es más flexible y potente.
Animal* → Perro obj
→ hablar()
ejecuta Perro::hablar()
Animal* → Gato obj
→ hablar()
ejecuta Gato::hablar()
Animal* → Vaca obj
→ hablar()
ejecuta Vaca::hablar()

¿Cómo funciona la vtable?

Cuando una clase tiene métodos virtual, el compilador genera una tabla de punteros a funciones (vtable). Cada objeto guarda un puntero secreto (vptr) a su vtable.

📋 vtable de Perro:
[0] hablar → Perro::hablar()
[1] moverse → Perro::moverse()
📋 vtable de Gato:
[0] hablar → Gato::hablar()
[1] moverse → Gato::moverse()
Cuando escribes a->hablar():
1. El CPU accede al objeto en memoria
2. Lee el vptr (puntero a su vtable)
3. Busca la entrada [0] (hablar)
4. Salta a esa función y la ejecuta

Todo esto ocurre en nanosegundos.

⚠️ Destructor Virtual

Si usas polimorfismo con memoria dinámica (new/delete), siempre declara el destructor como virtual en la clase base:

virtual ~Animal() {} // ✓
~Animal() {}         // ✗ memory leak

Clases Abstractas y Funciones Virtuales Puras

Una función virtual pura se declara con = 0.
Hace que la clase sea abstracta → no se puede instanciar. 
// Clase ABSTRACTA — no instanciable
class Shape {
public:
  virtual string draw() = 0; // pura
  virtual double area() = 0; // pura
  virtual ~Shape() {}
};

// Clase CONCRETA — sí instanciable
class Circle : public Shape {
private:
  double radio;
public:
  string draw() override {
    return "Dibujando círculo";
  }
  double area() override {
    return 3.14159 * radio * radio;
  }
};

✗ Esto da error de compilación:

Shape s; // ERROR
// "cannot instantiate
//  abstract class"

Shape es un contrato, no un objeto real. Nadie puede "ser solo un Shape".

✓ Esto funciona perfecto:

Shape* s = new Circle();
cout << s->draw();
// "Dibujando círculo"

Un puntero a Shape sí puede existir, apuntando a un objeto concreto.

Regla: Si una clase hija NO implementa todas las funciones puras de la base, también se vuelve abstracta y tampoco puede instanciarse.

El problema sin polimorfismo

✗ Sin polimorfismo — 30,000 líneas

// Para 5 figuras ya es repetitivo:
Circle    c1, c2, c3, c4;
Rectangle r1, r2, r3;
// ...y para 10,000 figuras?

c1.draw();
c2.draw();
c3.draw();
r1.draw();
r2.draw();
// ← 10,000 líneas más...

Código frágil, difícil de mantener, inescalable.

✓ Con polimorfismo — 3 líneas

Shape* figuras[10000];
// Llenar con cualquier tipo...
figuras[0] = new Circle();
figuras[1] = new Rectangle();
figuras[2] = new Polygon();
// ... 9997 más ...

// Dibujar TODAS en 3 líneas:
for (int i = 0; i < 10000; i++)
  figuras[i]->draw();
// ← Cada una se dibuja sola ✓

Escalable, flexible, fácil de extender.

Principio Open/Closed: El ciclo for nunca cambia. Para agregar un nuevo tipo de figura, solo crea una nueva clase que herede de Shape.

Caso Real: Sistema de Pagos (FinTech)

Empresas como Stripe, Mercado Pago o BBVA manejan múltiples métodos de pago. El polimorfismo es la solución estándar de la industria.

💳
Tarjeta de Crédito
SalariadoConComision: base + % ventas
🏦
Transferencia Bancaria
Empleado asalariado: sueldo fijo mensual
⏰
Pago por Horas
Freelancer: horas × tarifa
🔨
Pago por Destajo
Manufactura: piezas × precio_pieza
class Empleado { // abstracta
protected:
  string nombre;
public:
  virtual double calcularPago() = 0;
  virtual ~Empleado() {}
};

class Asalariado : public Empleado {
  double salarioMensual;
public:
  double calcularPago() override {
    return salarioMensual;
  }
};

class PorHoras : public Empleado {
  double horas, tarifaHora;
public:
  double calcularPago() override {
    return horas * tarifaHora;
  }
};

// Procesar nómina de 30,000 empleados:
vector<Empleado*> nomina = cargarNomina();
double total = 0;
for (auto* emp : nomina)
  total += emp->calcularPago(); // ✓
Este patrón (Strategy + Polymorphism) es usado en producción por Stripe, PayPal, SAP y la mayoría de sistemas de nómina empresarial.

Código Completo — Shape con Polimorfismo

Shape.h
#pragma once
#include <string>
using namespace std;

class Shape {
protected:
  int x, y;
public:
  Shape() : x(0), y(0) {}
  Shape(int x, int y)
    : x(x), y(y) {}
  virtual string draw() = 0;
  virtual double area() = 0;
  int getX() { return x; }
  int getY() { return y; }
  virtual ~Shape() {}
};
Circle.h / .cpp
// Circle.h
#include "Shape.h"
class Circle : public Shape {
  double radio;
public:
  Circle(int x, int y,
         double r)
    : Shape(x,y), radio(r) {}
  string draw() override;
  double area() override;
};

// Circle.cpp
string Circle::draw() {
  return "⬤ Círculo";
}
double Circle::area() {
  return 3.14159 * radio * radio;
}
main.cpp
#include "Circle.h"
#include "Rectangle.h"
#include <vector>

int main() {
  vector<Shape*> figs;
  figs.push_back(
    new Circle(0,0,5));
  figs.push_back(
    new Rectangle(1,2,4,3));

  for (auto* f : figs) {
    cout << f->draw()
         << " área="
         << f->area() << "\n";
    delete f;
  }
  return 0;
}
// Output:
// ⬤ Círculo área=78.5
// ▬ Rectángulo área=12

Buenas prácticas: override y final

✗ Bug silencioso (sin override)

class Base {
public:
  virtual void hazAlgo(int x);
};

class Hija : public Base {
public:
  // Oops: firma diferente (double)
  void hazAlgo(double x); // 💀
  // No sobreescribe — crea fn. nueva
  // El polimorfismo NO funciona
};

Error difícil de detectar. El programa compila pero se comporta mal.

✓ Detectado en compilación (con override)

class Hija : public Base {
public:
  void hazAlgo(double x)
         override; // ← ERROR!
  // Compilador avisa:
  // "hazAlgo(double) does not
  //  override any function"
};

override hace que el compilador valide la firma. Error detectado antes de tiempo.

final: bloquear herencia

class Seguro final { ... };
// Nadie puede heredar de Seguro

void metodo() final;
// Nadie puede sobreescribir esto
CaracterísticaSin overrideCon overridefinal
Detecta errores de firma✗✓✓
Documenta intención✗✓✓
C++ 11+N/A✓✓

Polimorfismo Estático vs. Dinámico

Característica ⚡ Estático (Compile-Time) 🔀 Dinámico (Run-Time)
¿Cuándo se resuelve? Al compilar Al ejecutar
Mecanismo Sobrecarga de funciones/operadores Funciones virtuales + herencia
Velocidad Más rápido Ligero overhead (vtable)
Flexibilidad Limitada Alta — extensible sin cambios
¿Requiere herencia? No Sí
Palabra clave — (misma firma, params distintos) virtual, override
Uso típico Operaciones matemáticas, utilidades Sistemas de plugins, juegos, patrones de diseño

Ejemplo estático — Sobrecarga

int    max(int a, int b);
double max(double a, double b);
string max(string a, string b);
// Compilador elige según args

Ejemplo dinámico — Virtual

Casilla* c = new CasillaSerpiente();
c->aplicarEfecto(jugador);
// → jugador retrocede 5 casillas
// Decidido en runtime

Código guía para la Tarea 2

Casillas (Tile.h)
class Casilla {      // abstracta
protected:
  int posicion;
public:
  Casilla(int pos) : posicion(pos) {}
  virtual int aplicarEfecto(
    int posActual) = 0;
  virtual ~Casilla() {}
};

class CasillaNormal : public Casilla {
public:
  CasillaNormal(int pos)
    : Casilla(pos) {}
  int aplicarEfecto(int pos)
      override {
    return pos; // sin cambio
  }
};

class CasillaSerpiente : public Casilla {
  int penalty;
public:
  CasillaSerpiente(int pos, int p)
    : Casilla(pos), penalty(p) {}
  int aplicarEfecto(int pos)
      override {
    cout << "🐍 Serpiente! -"
         << penalty << "\n";
    return pos - penalty;
  }
};
GameMode.h + uso en tablero
class GameMode {    // abstracta
public:
  virtual void ejecutar() = 0;
  virtual void manejarTurno() = 0;
  virtual ~GameMode() {}
};

class ModoManual : public GameMode {
public:
  void ejecutar() override {
    // espera input del usuario
  }
  void manejarTurno() override { ... }
};

// En el juego principal:
Casilla* tablero[100];
// inicializar casillas...
tablero[0] = new CasillaNormal(0);
tablero[5] = new CasillaSerpiente(5,3);
tablero[9] = new CasillaEscalera(9,4);

// Al caer en una casilla:
posJugador = tablero[posJugador]
              ->aplicarEfecto(posJugador);
// ← No importa el tipo: funciona

Polimorfismo correcto

  • Clase base Casilla tiene método virtual puro
  • CasillaNormal, CasillaSerpiente, CasillaEscalera heredan y sobreescriben

  • Destructores virtuales en clases base

  • main recibe 8 argumentos por línea de comandos
  • Valores por defecto si no se pasan argumentos
  • Modo A funciona automáticamente hasta ganador o max turns
  • Modo M pide confirmación entre turnos
  • Diagrama de clases (class_diagram.png) incluido
  • README con instrucciones y discusión de OOP
  • Zip con nombre correcto: TC1030 - T2 - A[matrícula].zip
Trampa común: No basta con tener clases que hereden. Para que haya polimorfismo real, debes usar un arreglo/vector de Casilla* y llamar aplicarEfecto() sin saber el tipo concreto. Si usas if/switch para decidir el tipo de casilla, no hay polimorfismo.
